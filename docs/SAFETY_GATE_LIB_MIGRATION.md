# 安全闸迁移评估 —— 浏览器路线 → firstrade 库路线(给 Data 调研折叠)

> %Exec 供稿(2026-06-10),配合 Data 调研 firstrade Python 库替代 Playwright。
> 立场:**评估,非背书**。浏览器代码 + 其 129 项执行测试**原地冻结**,本评估不动一行。
> 一句话结论:四个安全闸里**两个原样移植、一个改写、一个是决策级红线缺口**——
> 唯一 go/no-go 是 `paper_account_marker`("只碰模拟盘")在库模式下能否成立(§2.1)。
> 凭据进程化曾是第二顾虑,**operator 已授权**(§2.2),降级为非阻断项。

## 1. 逐闸迁移矩阵

| 安全闸 | 现状(浏览器) | 库模式 | 判定 |
|---|---|---|---|
| **PAPER_ONLY 硬钉** | `safety.assert_paper_only()`:模块级 `Final` + 真金环境变量黑名单 | 与传输层无关,逐字移植 | ✅ **原样移植** |
| **kill-switch** | 文件信号 `data/execution/KILL`,每个页面动作前 `check()` | 机制原样移植;但**插入点要重写**:从"每个 guarded 页面原语前"改为"每个库 API 调用前"(登录/下单/查询都要) | ✅ 机制移植 / ⚠️ 调用点重写 |
| **audit_log** | append-only JSONL,凭据字段名硬拒,记动作语义 | 与传输层无关,逐字移植;**库模式下更关键**(没有可旁观的浏览器,审计是唯一可观测面) | ✅ **原样移植** |
| **guarded 原语** | `goto/click/type_human` = kill→选择器核验→人类节奏→动作→审计 | 无 DOM/无选择器:重写为"guarded API 包装器" = kill.check→assert_paper_only→(限速)→库调用→审计。**模式移植,选择器核验整层消失** | ⚠️ **重写**(模式可复用) |
| **选择器注册表 + verified:false 闸** | 17 项 CSS,未核验拒跑 | 库模式无 CSS,整层作废。其底层原则"不经未核验的接口动作"→映射为"**钉死库版本 + 核验它打哪个 endpoint/哪个账户**" | 🔄 **形态转换**(原则在,实现全换) |
| **反检测**(去 automation 标志/养熟 profile/抹 webdriver) | 对抗浏览器指纹风控 | API 无浏览器指纹,整套作废;**但风控换面不换里**:库走 HTTP,可能有**设备指纹/User-Agent/登录频率**风控(2192 类拒登可能复现在 API 层) | 🔄 作废→需新调研:API 层风控面 |
| **人类节奏 HumanPacer** | 模拟真人击键/点击防 bot 识别 | 击键建模无意义;**保留轻量限速**(API rate-limit + 人类下单频率,红线 6),丢弃键级建模 | ⚠️ 降级保留 |
| **broker_order_ref/确认回采** | 抓页面文本(脆) | 库直接返回结构化 order id/状态 → **比抓页面更硬**,ledger r3 设计的可空降级反而用不上(直接有) | ✅ 升级(ledger 不变) |

## 2. 红线考量(§2.1 是唯一 go/no-go;§2.2 已授权降级)

### 2.1 `paper_account_marker` —— "只碰模拟盘"在无浏览器下怎么落地

浏览器模式靠 DOM 元素 + 账号号匹配确认在模拟盘。库模式分三种情况,**Data 调研必须先回答是哪种**:

- **情况 A(最硬):库有账户类型参数 / 账户列表暴露模拟账户号。**
  → 比 DOM 更强:下单前断言 `account_id == 配置的模拟账户号`,不符直接 `ExecutionHalted` 拒单。
  把"只碰模拟盘"从"找得到某个页面元素"升级为"**API 目标账户号精确匹配白名单**",fail-closed。**这是库路线唯一能接受 PAPER_ONLY 的形态。**
- **情况 B(致命):库只打真实经纪账户 endpoint,Firstrade 模拟盘(paper trading)是独立 web 产品、无对应 API。**
  → 库里**没有模拟盘**,任何下单都是真金 → **直接违反红线 2**。此情况下库路线对本项目**不可用**,
  浏览器模拟盘路线必须保留。光是把这种库 import 进仓库,就已逼近红线 2 的"真实下单代码"——
  **dual-use 风险:firstrade-api 类库本质能下真单。**
- **情况 C(待证):库声称支持 paper,但账户隔离靠某个易错参数(如 dry-run flag / 环境字符串)。**
  → 接受的前提是该参数能被 `assert_paper_only` 同级硬钉(Final 常量 + 启动断言 + 每单复核),
  不能是"调用方记得传"。否则一次漏传 = 真金事故。

> **给 Data 的头号调研项**:firstrade 库**到底支不支持模拟盘账户**?账户类型怎么指定?
> 是否存在"账户号"可供下单前精确断言?答 B = 库路线对本项目否决;答 A = 可行且更硬;答 C = 看参数能否硬钉。

### 2.2 凭据进程化 —— operator 已授权(非阻断项)

- **浏览器模式**凭据从不进 Python 进程是最强形态;**库模式**登录 API 需程序化提供
  用户名/密码/2FA,凭据必进进程。这是姿态变化,但 **operator 已明确授权可进程序**
  (2026-06-10,Lead 转达),故**不再是 go/no-go 阻断项**。
- 落地仍守红线 5(密钥不入码/不入库),按既有纪律:凭据走 `.env`(gitignored)或系统 keychain,
  绝不硬编码、绝不提交;`audit_log` 的凭据字段名硬拒从"选择器名"扩到"库参数名"(沿用现有黑名单机制)。
- 2FA(邮箱/SMS/PIN OTP)程序化获取是新面,实现时单列设计;不影响可行性判定。

## 3. 给 Data 的净结论

1. **能原样搬的**(省事):PAPER_ONLY 硬钉、真金环境变量黑名单、kill-switch 文件机制、audit_log。这四块与传输无关,库路线直接复用 `src/execution/safety.py` + `audit_log.py`,**别重写**。
2. **改写但有样板**:把 guarded 页面原语的"kill→核验→限速→动作→审计"五步模式,搬成 guarded API 包装器;kill 检查插到每个库调用前。
3. **作废**:选择器注册表/核验、浏览器反检测、键级人类节奏、profile 养熟——库模式不需要(但风控可能在 API 层换面复现,需新调研)。
4. **唯一 go/no-go(Data 在查死)**:firstrade 库**是否支持模拟盘账户**、能否暴露账户号供下单前精确断言?(情况 A/B/C)——**答 B 则库路线对 PAPER_ONLY 否决。** 凭据进程化已由 operator 授权,不再是阻断项(§2.2)。
5. **倾向**(诚实声明,非结论):库模式在 broker_order_ref/结构化状态上更干净、省掉浏览器稳定性风险;凭据进程化已授权,故**可行性收敛到单一前提:情况 A 是否成立**。① 未证实前,浏览器模拟盘路线不应拆除——它是当前唯一已证能保证"只碰模拟盘"的路线。

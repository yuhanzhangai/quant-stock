# 总体路线

你的下一阶段按这个顺序推进：

```text
Checkpoint 0  冻结当前基线
Checkpoint 1  整理项目结构和策略状态
Checkpoint 2  建立 research database
Checkpoint 3  建立 data manifest
Checkpoint 4  建立 data quality gate
Checkpoint 5  标准化 backtest 输出
Checkpoint 6  建立 experiment ledger
Checkpoint 7  统一 validation pipeline
Checkpoint 8  加入成本、滑点、风险压力测试
Checkpoint 9  策略准入门禁
Checkpoint 10 Paper trading 重构
Checkpoint 11 Dashboard 升级为研究控制台
Checkpoint 12 单人长期工作流
```

核心原则：

```text
一次只做一个 checkpoint
每个 checkpoint 必须有明确产出物
没有产出物就不算完成
没有通过标准就不能进入下一步
策略结果不能只看 Sharpe
任何结果必须能复现
```


### 4.2 实现基础检查

先实现这几个，不要一开始做太复杂：

```text

check_missing_bars
check_ohlc_validity

```

---

### 4.3 每个检查的含义

#### duplicate timestamps

检查同一个 symbol/timeframe 是否有重复 timestamp。

失败例子：

```text
ETH-USDT 5m 2026-02-01 10:00:00 出现两行
```

严重程度：

```text
critical
```

处理方式：

```text
回测拒绝运行
```

---

#### timestamp order

检查 timestamp 是否单调递增。

严重程度：

```text
critical
```

---

#### missing bars

检查是否缺 K 线。

例如 5m 数据：

```text
10:00
10:05
10:10
10:20
```

缺了：

```text
10:15
```

严重程度：

```text
warning 或 critical
```

规则建议：

```text
小缺口：warning，允许回测但记录
大缺口：critical，拒绝回测
```

---

#### OHLC validity

检查：

```text
high >= open
high >= close
high >= low
low <= open
low <= close
open > 0
high > 0
low > 0
close > 0
```

严重程度：

```text
critical
```

---

#### volume validity

检查：

```text
volume >= 0
volume 不应该是 null
```

严重程度：

```text
warning / critical
```

---

#### price jump

检查异常跳价。

例如：

```text
单根 5m K线 close 变化超过 30%
```

不一定是错，可能是黑天鹅，但必须标记。

严重程度：

```text
warning
```

---

#### latest bar delay

用于 live / paper trader。

检查：

```text
当前时间 - 最新K线时间 > 2个 timeframe
```

严重程度：

```text
critical for live signal
warning for research
```

### 7.2 定义 Gate 列表

### 7.3 每个 Gate 输出统一格式

```json

  "gate_name": "walk_forward",
```

```text

warning
```

---

### 7.4 Gate 1：data_quality

规则：

```text
critical issue count = 0
```

失败则直接停止后续验证。

---

### 7.5 Gate 2：baseline_backtest

规则示例：

```text
trade_count >= 30
profit_factor > 1.1
max_drawdown > -0.30
avg_trade_return > 0
```

注意：baseline 不要求特别高，只要求策略不是明显垃圾。

---

### 7.6 Gate 3：cost_stress

至少三档：

```yaml
cost_stress:
  optimistic:
    fee_multiplier: 1.0
    slippage_multiplier: 0.5
  normal:
    fee_multiplier: 1.0
    slippage_multiplier: 1.0
  pessimistic:
    fee_multiplier: 1.5
    slippage_multiplier: 2.0
```

通过标准：

```text
normal 成本下必须通过
pessimistic 成本下不能崩溃
```

建议规则：

```text
normal_profit_factor > 1.25
pessimistic_profit_factor > 1.0
```

---

### 7.7 Gate 4：out_of_sample

规则：

```text
OOS profit factor > 1.25
OOS trade count >= 30
OOS max drawdown > -0.25
```

如果交易次数少于 30，不要直接 accepted，最多是 warning 或 inconclusive。

---

### 7.8 Gate 5：walk_forward

规则：

```text
positive_windows_ratio >= 0.60
median_window_profit_factor > 1.1
worst_window_drawdown > -0.25
```

对于你这种单人、小账户、加密 5m 策略，我不建议门槛太完美。先要求“多数窗口能活下来”。

---

### 7.9 Gate 6：random_baseline

至少比较：

```text
random entry + 同样交易频率
random entry + 同样持仓时间
反向信号
buy and hold
no trade
```

通过标准：

```text
strategy_expectancy > random_expectancy_p75
strategy_profit_factor > random_profit_factor_p75
strategy_max_drawdown 不明显更差
```

意思是：你的策略至少要打败随机策略的上四分位，而不是只打败随机平均值。

---

### 7.10 Gate 7：monte_carlo

规则：

```text
盈利概率 > 60%
最差 5% 回撤不超过账户可接受范围
risk_of_ruin 接近 0
```

对于你的 $50 小账户，重点不是中位收益，而是：

```text
最差 5% 场景会不会把账户打残
```

---

### 7.11 Gate 8：event_backtest

检查重大事件期间表现：

```text
LUNA
FTX
ETF
极端暴跌
极端上涨
高波动窗口
```

通过标准不是必须赚钱，而是：

```text
不能出现系统性失控
不能出现远超正常分布的亏损
```

---

### 7.12 Gate 9：parameter_stability

检查参数是不是只有一个点有效。

例如：

```text
trend_ma = 180 很好
trend_ma = 170 / 190 全部崩溃
```

这种大概率是过拟合。

通过标准：

```text
最佳参数附近存在一片可接受区域
```

# Checkpoint 12：单人长期工作流

## 目标

你的系统已经复杂，如果没有流程，很容易陷入：

```text

```

这个 checkpoint 是给你自己建立“纪律系统”。

---

## 每日检查清单

每天只看这些，不做深度研究：

```text
[ ] 数据是否更新
[ ] 最新 K 线是否延迟
[ ] OKX API 是否正常
[ ] data_quality 是否有 critical
[ ] production 策略是否有异常信号
[ ] paper trader 是否正常记录
[ ] 是否有重复信号
[ ] 是否触发风险限制
[ ] Telegram 通知是否正常
```

产出：

```text
reports/daily_health/YYYYMMDD.json
```

---

## 每轮研究清单

每次只允许一个主实验。

开始前：

```text
[ ] 写 experiment.yml
[ ] 写 hypothesis
[ ] 固定数据范围
[ ] 固定参数搜索范围
[ ] 固定 success criteria
[ ] 确认没有使用测试集调参
```

运行中：

```text
[ ] 记录 run_id
[ ] 记录 data_version
[ ] 记录 code_commit
[ ] 保存 config
[ ] 保存 metrics
[ ] 保存 trades
[ ] 保存 equity
```

结束后：

```text
[ ] 写 conclusion
[ ] accepted / rejected / inconclusive 三选一
[ ] 更新 strategy card
[ ] 更新 registry/strategies.yml
[ ] 如果 rejected，移动到 experiments/rejected/
[ ] 如果 accepted，进入 candidate，不直接 production
```

---
